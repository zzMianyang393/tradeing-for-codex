## 四、下一阶段正确任务

### 任务 1：先由 Claude Code 提出 3 个新策略候选

候选必须是逻辑上独立的新策略，不得只是给旧策略增加 RSI、成交量、均线阈值。

每个候选必须说明：

1. 利润来源假设是什么。
2. 在什么行情中应该有效。
3. 为什么已有策略没有覆盖它。
4. 需要哪些数据。
5. 是否可以用当前 15m 数据完成无未来泄漏回测。
6. 可能的失效机制是什么。

优先考虑以下策略族：

- 跨币种相对强弱/弱势轮动，而不是单币绝对指标。
- 多周期确认策略，例如 4h 状态 + 15m 入场，但必须处理 BTC/ETH 数据覆盖差异。
- 事件后的波动衰减或反转策略，要求明确事件定义和净 PnL 标签。
- 资金费率策略只能作为 90 天研究候选，不能作为 365 天结论，除非找到可信的历史数据源。

### 任务 2：实现无参数搜索的策略基线

每个候选只实现一套保守默认规则，输出：

- 90 天、180 天、365 天收益。
- 最大回撤、胜率、盈亏比、交易次数。
- 12 个月拆分。
- `by_symbol`、`by_reason`、`by_regime`。
- 固定币种集合和全部币种结果。
- 手续费、滑点后的净 PnL。

禁止先做参数网格搜索。

### 任务 3：固定币种宇宙

至少测试以下宇宙：

1. BTC/ETH/SOL/BNB/XRP 主流集合。
2. 固定 5 个高流动性山寨币集合。
3. 全部可用 15m 币种。

如果策略只在一个宇宙盈利，标记为“币种集合敏感”，不得进入优化。

### 任务 4：硬性淘汰标准

任何候选满足以下任一条件，直接淘汰：

- 365 天亏损。
- 180 天和 90 天同时亏损。
- 全部币种与主流币集合方向相反。
- 单个月份或少数交易贡献大部分利润。
- 最大回撤超过 45%。
- 交易次数太少，无法形成统计证据。
- 需要针对某个月份、某几个币、某个窗口单独改参数。

## 五、只有通过基线后才允许做的事

只有新策略通过无参数基线，才允许：

1. 做小范围参数敏感性测试，参数最多 3 个。
2. 做滚动 walk-forward。
3. 做 12 个月稳定性分析。
4. 与其他已通过策略组合。
5. 进入模拟盘观察。

模拟盘不是为了证明一个明显亏损策略会赚钱，而是验证执行、滑点、订单状态、对账和暂停机制。

## 六、当前不要做什么

- 不要恢复旧的 CP MFE 标签。
- 不要继续调 `transition_breakout_long`、`trend_short_factor` 或 `range_revert_short` 的参数。
- 不要把动态行情路由当成收益来源。
- 不要使用单一 365 天窗口的高收益报告。
- 不要为了达到 10U/30天或 365天目标而提高杠杆和风险比例。
- 不要把未通过真实净 PnL 验证的策略接入模拟盘自动下单。

Claude Code 下一次交付应包含：

## 八、旧候选阶段结果已作废（2026-07-10 更新）

此前的三个候选结果不能再作为独立策略结论。审计发现候选脚本经常只统计自己的条件，实际回测却调用旧动态路由；`unified_validation.py` 还曾直接让量价背离复用日内反转配置。

以下是旧验证的历史记录，不能用于淘汰或接受策略：

1. 相对强弱轮动：365 天亏损，淘汰。
2. 多周期确认：全量 365 天约 `+2.73%`，但剔除 2026-05 至 2026-07 后约 `-5.15`，标记为高过拟合风险，禁止进入模拟盘。
3. 波动率压缩突破：365 天亏损，淘汰。

新的统一验证器已改为调用 `candidate_strategies.py` 中的真实入场信号，并记录 `strategy_id`、实现路径与交易指纹。第一份真实冒烟报告为 `intraday_reversal`：365 天主流币净收益约 `+6.70%`，但压力成本后约 `-18.52%`，且仅 4 笔交易；不合格、不进入模拟盘。这只是证明新验证器已生效，不是策略成功。

因此当前仍未发现可验证的正期望策略。下一步禁止继续扩大参数搜索，应该转为：

- 核对三个候选的原始报告和收益口径。
- 修复数据版本、币种宇宙和费用假设的统一记录。
- 研究新的数据来源或新的市场结构假设。
- 在没有新证据前，不进入模拟盘和实盘。

所有新候选必须先用 `--windows 90 --universes majors` 做真实信号冒烟验证，再逐步扩大到 90/180/365 天和三个宇宙；不要启动无法观察进度的大矩阵任务。
