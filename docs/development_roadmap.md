# 量化交易系统后续开发路线图

这份文档用于约束后续开发方向：当前项目不能只追求某一张回测报告的漂亮数字，而要逐步从“K线策略回测框架”升级成“可模拟盘验证的自动交易系统”。

## 当前判断

当前程序已经具备本地 K 线回测、分窗口配置、滚动审计和基础风控，但还不是成熟量化交易系统。

主要问题：

- 信号来源偏少，主要依赖 K 线指标。
- 交易机会密度不足，短周期收益低。
- 高收益配置容易在滚动窗口里暴露过拟合。
- 还没有模拟盘执行、订单状态、持仓对账和异常恢复。
- 风控更多服务于回测，还没有完整实盘风控闭环。

后续目标不是单纯把 30 天、90 天收益调高，而是建立一套能持续验证、能模拟执行、能控制亏损的完整系统。

## 长期目标

把项目升级为：

1. 多数据源量化研究系统。
2. 多策略组合回测系统。
3. 带硬性反过拟合验证的策略筛选系统。
4. 可连接模拟盘的自动交易执行系统。
5. 有风控、监控、日志、报告和暂停机制的实盘预备系统。

## 一、数据层

当前主要依赖本地 K 线。成熟系统需要扩展数据源。

需要增加：

- 1m、5m、15m、1h 多周期 K 线。
- 实时 ticker。
- 成交明细 trades。
- 主动买入、主动卖出量。
- 订单簿深度 order book。
- 资金费率 funding rate。
- 持仓量 open interest。
- 爆仓数据 liquidation。
- 多交易所价差。
- BTC、ETH 与山寨币联动强弱数据。

怎么做：

- 建立统一数据 schema。
- 数据采集器和回测读取器使用同一套结构。
- 对每个数据源记录延迟、缺失和异常。
- 本地优先使用 Parquet 或 SQLite，后续再考虑 Postgres。
- 所有回测报告记录数据版本和时间范围。

预计周期：1 到 2 周。

验收标准：

- 数据下载、增量更新、完整性检查可重复执行。
- 任意策略可以读取相同的数据接口。
- 报告中能显示数据缺失率和可用时间跨度。

## 二、策略层

当前策略偏单一。成熟系统应是多策略组合。

需要开发的策略族：

- 均值回归策略：保留当前表现较好的反转逻辑。
- 趋势跟随策略：专门处理单边行情。
- 短线反抽/回踩策略：用于提升 7、14、30 天机会密度。
- 波动率突破策略：只在波动率状态合适时启用。
- 资金费率策略：利用 funding rate 偏离。
- BTC 强弱联动策略：识别山寨币相对 BTC 的强弱。
- 极端下跌反弹策略：只在恐慌释放后低频介入。
- 多币种轮动策略：动态选择近期 edge 更强的币种。

怎么做：

- 每个策略输出统一 Signal。
- Signal 必须包含 strategy、reason、direction、score、risk_hint。
- 策略只负责产生信号，不直接决定仓位。
- 组合层根据风险预算、相关性和近期表现决定是否下单。

预计周期：3 到 6 周。

验收标准：

- 每个策略都有独立回测报告。
- 每个策略都能按 symbol、reason、regime 拆账。
- 任一策略不能靠单笔交易贡献大部分收益。

## 三、验证层

这是防止过拟合的核心。

必须建立的验证：

- walk-forward 测试。
- rolling-window 审计。
- 样本内、样本外切分。
- 参数敏感性测试。
- 单笔贡献占比检查。
- 分币种表现。
- 分市场状态表现。
- 手续费、滑点、资金费率模拟。
- 蒙特卡洛扰动。
- 回撤压力测试。

硬性原则：

- 最新窗口收益高，不代表策略可用。
- 滚动切片不过关，不能进入默认配置。
- 如果某个窗口收益来自 1 到 2 笔交易，必须降级为疑似过拟合。
- 如果相邻参数表现差异巨大，必须降级为疑似过拟合。

建议初始闸门：

- 7、14、30 天滚动盈利率至少 70%。
- 60、90、180 天滚动盈利率至少 75%。
- 任一滚动切片最差收益不应出现爆仓式亏损。
- 单笔最大正收益占总正收益不超过 70%。
- 单窗口最大回撤建议不超过 45%，越低越好。

预计周期：2 到 4 周。

验收标准：

- 每次接受新配置时，同时生成 latest report 和 rolling audit。
- 报告必须明确标记 PASS、CHECK、FAIL。
- 不能只用单次回测结果作为合并依据。

## 四、风控层

成熟量化系统的核心不是信号，而是风控。

需要增加：

- 单币种最大仓位。
- 总仓位限制。
- 策略级风险预算。
- 单日最大亏损。
- 单周最大亏损。
- 连亏暂停。
- 波动率异常暂停。
- 滑点过大暂停。
- 交易所异常暂停。
- 强平距离保护。
- 相关性限仓。
- 自动降杠杆。
- 亏损后动态降低风险。

怎么做：

- 所有订单必须先经过 RiskManager。
- RiskManager 有权拒单、减仓、暂停策略。
- 每个策略独立统计 PnL、回撤和连亏。
- 风控事件必须写入日志和报告。

预计周期：2 到 3 周。

验收标准：

- 任意下单前都能看到通过或拒绝的原因。
- 模拟盘中出现异常时，系统能自动暂停。
- 本地状态与交易所持仓不一致时禁止继续开仓。

## 五、执行层

回测赚钱不代表实盘可交易。执行层必须单独建设。

需要增加：

- 模拟盘交易接口。
- 实盘交易接口预留。
- 下单失败重试。
- 订单状态同步。
- 部分成交处理。
- 撤单逻辑。
- 滑点保护。
- 限价单、市价单策略。
- 最小下单量处理。
- 断线恢复。
- 持仓对账。
- 交易日志。

怎么做：

- 先接 OKX 或 Binance 模拟盘。
- 所有订单写入本地数据库。
- 周期性同步交易所真实持仓。
- 本地状态和交易所状态不一致时暂停交易。
- 执行层和策略层解耦，策略只发目标意图。

预计周期：2 到 4 周。

验收标准：

- 可以在模拟盘连续运行至少 2 到 4 周。
- 订单、成交、持仓、资金变化都能追溯。
- 网络断开、下单失败、部分成交都能被正确处理。

## 六、监控与运维层

系统跑起来后，必须能知道它为什么赚钱、为什么亏钱、什么时候该停。

需要增加：

- 实时权益曲线。
- 当前持仓。
- 当前风险暴露。
- 今日盈亏。
- 每个策略盈亏。
- 最近错误。
- 交易通知。
- 自动暂停和恢复状态。
- 每日报告。
- 每周复盘报告。

怎么做：

- 先做 CLI 报告。
- 后续再做简单 Web dashboard。
- 日志结构化。
- 异常通过 Telegram、邮件或其他通知渠道发送。

预计周期：1 到 2 周。

验收标准：

- 每天能自动生成交易总结。
- 异常能主动通知。
- 任意一笔交易都能追溯到信号、风控和订单。

## 推荐开发顺序

### 阶段 1：先补真实系统骨架

目标：从回测程序升级为可模拟盘验证的系统。

任务：

1. 建立订单、成交、持仓、账户状态模型。
2. 实现模拟盘执行器。
3. 实现 RiskManager 下单前置检查。
4. 统一交易日志。
5. 输出每日模拟盘报告。

预计周期：2 到 4 周。

### 阶段 2：扩展数据源

目标：不要只靠 K 线指标找信号。

任务：

1. 增加 funding rate。
2. 增加 trades 主动买卖量。
3. 增加 order book snapshot。
4. 增加 open interest。
5. 把数据源接入回测特征。

预计周期：1 到 2 周。

### 阶段 3：策略组合

目标：从单策略变成多策略组合。

任务：

1. 抽象 Strategy 接口。
2. 保留当前均值回归策略。
3. 新增短线回踩/反抽策略。
4. 新增 funding 策略。
5. 新增 BTC 强弱联动策略。
6. 建立组合层。

预计周期：3 到 6 周。

### 阶段 4：反过拟合验证常态化

目标：任何配置进入默认前都必须过硬闸门。

任务：

1. 完善 rolling audit。
2. 增加 walk-forward。
3. 增加参数敏感性测试。
4. 增加单笔贡献占比检查。
5. 报告里明确 PASS、CHECK、FAIL。

预计周期：2 到 4 周。

### 阶段 5：模拟盘长期运行

目标：让系统在真实交易环境里暴露问题，但不冒真实资金风险。

任务：

1. 模拟盘连续运行 2 到 4 周。
2. 每日复盘交易、滑点、异常。
3. 调整执行层和风控层。
4. 只在模拟盘稳定后考虑小资金实盘。

预计周期：1 到 2 个月。

## 总体时间预估

在全部借助 AI 开发、用户持续反馈和测试的前提下：

- 最小可用模拟盘版本：2 到 4 周。
- 比较完整的成熟雏形：2 到 3 个月。
- 值得小资金实盘验证：3 到 6 个月。
- 稳定运行并持续优化：6 到 12 个月。

AI 可以加速代码、测试、报告和分析，但不能跳过市场验证周期。

## 后续开发原则

1. 不再只追单张回测表。
2. 高收益必须同时通过 rolling audit。
3. 默认配置不能靠某个窗口特化撑起来。
4. 新策略必须先独立验证，再进入组合。
5. 模拟盘执行框架优先级高于继续调参数。
6. 风控层优先级高于收益最大化。
7. 所有自动下单逻辑必须可追溯、可暂停、可复盘。

## 最近优先级

下一步建议按这个顺序做：

1. 建立模拟盘执行模型。
2. 建立 RiskManager。
3. 建立订单、成交、持仓日志。
4. 扩展 funding rate 和主动买卖量数据。
5. 开发短线回踩/反抽均值回归策略。
6. 把 rolling audit 作为默认验收闸门。

当前项目不应继续只靠 K 线参数搜索来追收益。要继续变成熟，必须进入“数据、策略组合、风控、执行、监控”的系统化建设阶段。
